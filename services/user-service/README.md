# Survey Platform API — User Service

## 1. Название и назначение сервиса

User Service — микросервис управления пользователями для платформы создания опросов. Сервис отвечает за регистрацию, аутентификацию, предоставление информации о пользователях, а также **геймификацию** (начисление XP, уровни, таблица лидеров).

**Основные функции:**
- Регистрация новых пользователей
- Аутентификация с выдачей JWT-токенов
- Получение информации о пользователе по ID
- **Начисление XP за действия (например, создание ответа)**
- **Расчёт уровня пользователя (100 XP = 1 уровень)**
- **Получение статистики пользователя (XP, уровень)**
- **Таблица лидеров (сортировка по убыванию XP)**
- **Приём внутренних событий от других микросервисов с защитой от дублирования**
- Проверка работоспособности сервиса

## 2. Архитектура и зависимости

### Технологии и фреймворки

- **Python 3.11+** — язык программирования
- **FastAPI** — веб-фреймворк для построения API
- **Uvicorn** — ASGI-сервер для запуска приложения
- **SQLite** — встраиваемая реляционная база данных
- **JWT (HS256)** — механизм аутентификации
- **PBKDF2-SHA256** — алгоритм хеширования паролей
- **Pydantic** — валидация данных и сериализация
- **unittest + TestClient** — тестирование API
- **Ruff** — линтинг и форматирование кода
- **Alembic** — миграции базы данных

### Взаимодействие с другими микросервисами

Сервис принимает **внутренние события** от сервиса опросов (Survey Service) через защищённый эндпоинт `/internal/events/answer-created`. В будущем также возможна интеграция с Analytics Service и Survey Service.

### Используемые внешние сервисы

- **SQLite** — локальное хранение данных (в продакшене планируется замена на PostgreSQL)
- Внешние сервисы (S3, Redis, Kafka) на данном этапе не используются

## 3. Способы запуска сервиса

### Локальный запуск (без Docker)

```bash
# 1. Установить зависимости
pip install -r requirements.txt
pip install -r requirements-dev.txt   # для тестов и линтера

# 2. Установить переменную окружения для внутреннего API-ключа
export INTERNAL_API_KEY="internal-test-key"   # Linux/macOS/Git Bash
# или
set INTERNAL_API_KEY=internal-test-key        # Windows CMD
# или
$env:INTERNAL_API_KEY = "internal-test-key"   # Windows PowerShell

# 3. Запустить сервер
./.venv/Scripts/python.exe -m uvicorn app.main:app --app-dir src --reload --port 8001

# 4. Открыть документацию в браузере
# http://127.0.0.1:8001/docs
```
Важно: Переменная INTERNAL_API_KEY обязательна для работы внутреннего эндпоинта. В тестах используется значение internal-test-key.

## Основные эндпоинты

| Эндпоинт | Метод | Описание | Требует авторизацию |
|----------|-------|----------|---------------------|
| /health | GET | Проверка работоспособности сервиса | Нет |
| /register | POST | Регистрация нового пользователя | Нет |
| /login | POST | Аутентификация, получение JWT-токена | Нет |
| /users/{id} | GET | Получение данных пользователя по ID | Да (JWT) |
| /users/{id}/stats | GET | Получение статистики (XP, уровень) | Нет (публичный) |
| /leaderboard | GET | Таблица лидеров (сортировка по XP) | Нет (публичный) |
| /internal/events/answer-created | POST | Внутренний эндпоинт для начисления XP | Да (внутренний API-ключ) |

# 5. Геймификация и обработка событий

## Модель пользователя

В таблицу users добавлены поля:

- `xp` (INTEGER) — очки опыта (по умолчанию 0)
- `level` (INTEGER) — уровень пользователя (по умолчанию 0)

Уровень вычисляется как: `level = xp // 100` (целочисленное деление).

## Начисление XP

XP начисляется при получении события `answer.created` от сервиса опросов. За каждое событие пользователь получает +5 XP. Сервис гарантирует, что одно и то же событие (по уникальному `answer_id`) не будет обработано дважды.

## Защита от дублирования (дедупликация)

Реализованы два уровня защиты:

1. По `answer_id` — в таблице `processed_events` создаётся уникальный индекс на (`user_id`, `answer_id`). Повторная отправка события с тем же `answer_id` для того же пользователя не приводит к повторному начислению XP.
2. По `Idempotency-Key` — если клиент передаёт заголовок `Idempotency-Key`, сервер сохраняет его в таблицу `idempotency_keys` и при повторных вызовах с тем же ключом возвращает ранее сохранённый ответ.

## Состояние операции

Каждое событие проходит статусы:

- `pending` — начата обработка
- `completed` — успешно начислены XP
- `failed` — произошла ошибка (например, пользователь не найден)

## Защита внутреннего эндпоинта

Эндпоинт `/internal/events/answer-created` доступен только при наличии корректного заголовка:

- `X-Internal-Token: <ключ>` или
- `Authorization: Bearer <ключ>`

Ключ задаётся переменной окружения `INTERNAL_API_KEY`. Без правильного ключа сервер возвращает `401 Unauthorized`.

# 6. Примеры запросов

### 1. Регистрация пользователя

```bash
curl -X POST http://127.0.0.1:8001/register \
  -H "Content-Type: application/json" \
  -d '{"email": "user@example.com", "password": "StrongPass123"}'
```

### 2. Вход в систему

```bash
curl -X POST http://127.0.0.1:8001/login \
  -H "Content-Type: application/json" \
  -d '{"email": "user@example.com", "password": "StrongPass123"}'
```

### 3. Получение информации о пользователе
```bash
curl -X GET http://127.0.0.1:8001/users/1 \
  -H "Authorization: Bearer <ваш_токен>"
```

### 4. Получение статистики (XP, уровень)
```bash
curl http://127.0.0.1:8001/users/1/stats
``` 
Ответ:

```json
{
  "id": 1,
  "email": "user@example.com",
  "xp": 125,
  "level": 1
}
```

### 5. Таблица лидеров
```bash
curl "http://127.0.0.1:8001/leaderboard?limit=10&offset=0"
```

Ответ: список пользователей, отсортированный по убыванию XP.

### 6. Начисление XP (внутреннее событие)
```bash
curl -X POST http://127.0.0.1:8001/internal/events/answer-created \
  -H "Content-Type: application/json" \
  -H "X-Internal-Token: internal-test-key" \
  -H "Idempotency-Key: 123e4567-e89b-12d3-a456-426614174000" \
  -d '{"user_id": 1, "answer_id": 101, "question_id": 5, "survey_id": 10}'
```

Ответ: обновлённая статистика пользователя.

## 8. Миграции базы данных (Alembic)

Для управления схемой БД используется Alembic. Миграции находятся в папке `alembic/versions/`.

#### Применение миграций на чистой базе

```bash
# Удалить старую БД (если есть)
rm data/survey_platform.db

# Применить все миграции
alembic upgrade head
```
Доступные миграции
0001_initial.py — создание таблицы users

0002_add_xp_and_level_to_users.py — добавление полей xp, level

0003_create_event_deduplication_tables.py — создание таблиц processed_events, idempotency_keys

0004_fix_unique_constraint_events.py
